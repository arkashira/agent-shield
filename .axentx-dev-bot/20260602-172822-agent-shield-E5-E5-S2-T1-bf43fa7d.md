# Contributing to **agent‑shield**

Thank you for your interest in improving **agent‑shield**!  
Please read the guidelines below before opening a Pull Request (PR).

## 📦 Setup Instructions

1. **Clone the repo**  
   ```bash
   git clone https://github.com/axentx/agent-shield.git
   cd agent-shield
   ```

2. **Create a virtual environment** (recommended)  
   ```bash
   python -m venv venv
   source venv/bin/activate   # on Windows: venv\Scripts\activate
   ```

3. **Install dependencies**  
   ```bash
   pip install -r requirements.txt
   ```

4. **Run the test suite** to confirm everything works locally  
   ```bash
   python -m unittest discover -s tests
   ```

## 🧩 Coding Standards

| Area                | Requirement |
|---------------------|-------------|
| **Style**           | Follow **PEP 8**. Use `black` (line length 88) and `pycodestyle` for linting. |
| **Indentation**     | 4 spaces, no tabs. |
| **Naming**          | `snake_case` for functions/variables, `PascalCase` for classes. |
| **Docstrings**      | Google style docstrings for all public modules, classes, and functions. |
| **Type hints**      | Add type hints to all public signatures. |
| **Tests**           | 100 % coverage for new/changed code (use `coverage run …`). |
| **Dependencies**    | Add new runtime deps to `requirements.txt`; keep the file sorted alphabetically. |

## 🔀 PR Review Process

1. **Branch** – Create a short‑lived branch from `main`  
   ```bash
   git checkout -b <type>/<short-description>
   # e.g. feat/add‑memory‑cache   or   fix/typo‑readme
   ```

2. **Commit** – Write clear, atomic commits. Prefix the subject with a conventional commit type (`feat:`, `fix:`, `docs:`, …).

3. **Reference an Issue** – Every PR **must** reference an open GitHub issue using one of the supported keywords (`Fixes #123`, `Closes #123`, `Resolves #123`). The reference can appear in the PR description or in the commit message.

4. **Push & Open PR** – Push the branch and open a PR against `main`. The PR template (see `.github/PULL_REQUEST_TEMPLATE.md`) will remind you of the required sections.

5. **Automated Checks** – CI will run automatically (see below). The PR cannot be merged until all checks pass.

6. **Review** – At least one maintainer must approve the PR. Reviewers may request changes; address them before merging.

7. **Merge** – Use the **Squash and merge** option to keep a linear history.

## ✅ Required GitHub Issue Reference

All PRs must be linked to an existing issue. The CI workflow (`issue‑link.yml`) will fail if no reference is found. Example:

```
Fixes #42 – Add caching layer for tool calls
```

## 🤖 CI Checks

The repository uses **GitHub Actions** to enforce:

* **Unit tests** (`python -m unittest discover -s tests`)
* **Code style** (`black --check .` and `pycodestyle .`)
* **Type checking** (`mypy .`)
* **Issue‑link validation** (`issue‑link.yml`)

If any job fails, the PR status will be **failed** and cannot be merged until the problem is resolved.
