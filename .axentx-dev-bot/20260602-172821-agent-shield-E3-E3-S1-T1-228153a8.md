### 1. Acceptance Criteria
1. **Complete Log Entry** – Every tool invocation creates a single JSON log line containing: `agent_id`, `tool_name`, `input` (max 1 KB), `output` (max 1 KB), `timestamp` (ISO‑8601 UTC), and `status` (`success`|`error`).
2. **Retention Policy** – Log files are written under `/var/log/agent‑shield/` and automatically rotated/removed so that no log older than 7 days remains on disk.
3. **Redaction** – Fields matching a configurable sensitive‑key list (e.g., `api_key`, `secret`, `password`) are replaced with the string `"[REDACTED]"` before being persisted.
4. **Size Truncation** – If `input` or `output` exceeds 1024 bytes, the stored value is truncated to the first 1024 bytes and suffixed with `"...[TRUNC]"`.
5. **JSON Validity** – Each log line is a well‑formed JSON object that can be parsed by standard JSON parsers (no trailing commas, proper escaping).
6. **Concurrency Safety** – Concurrent tool calls from multiple agents must not corrupt the log file (no interleaved JSON fragments).
7. **Performance Bound** – Logging adds ≤ 5 ms latency to a tool call on a typical workload (measured over 1000 calls).
