| # | Criterion | Measurable Target |
|---|-----------|-------------------|
| 1 | **Retry Count** | Tool calls that fail with a transient error (HTTP 5xx or timeout) are retried **exactly 3 times** before giving up. |
| 2 | **Back‑off Timing** | Each retry waits an exponentially increasing delay: 100 ms, 200 ms, 400 ms (i.e., `base * 2^(attempt-1)`). |
| 3 | **Configurable Max Retries** | The maximum number of retries is read from the env var `MAX_RETRIES`. If unset, defaults to **3**. |
| 4 | **Logging** | On each retry, a log entry is emitted containing: tool name, attempt number, error code/message, and timestamp. |
| 5 | **Circuit Breaker** | After **5 consecutive failures** for a specific tool, the breaker trips and blocks further calls for **60 s**. Subsequent calls during the block period immediately return a `CircuitBreakerOpen` error. |
| 6 | **Failure Propagation** | After the final retry or circuit breaker activation, the original error (or a wrapped `CircuitBreakerOpen` error) is returned to the caller. |
| 7 | **No Side‑Effects on Success** | A successful call (HTTP 2xx) should bypass any retry logic and return immediately. |
