## Unified, Production‑Ready Tracing Module

Below is a single, self‑contained implementation that pulls the best ideas from both candidates:

* **In‑memory store** – `DashMap` for lock‑free concurrent access.  
* **Full trace record** – agent ID, tool name, input/output (truncated), start/end timestamps, and a UUID trace ID.  
* **JSON logging** – `tracing` crate emits structured logs that can be consumed by any observability platform.  
* **Endpoint** – `/traces/<trace_id>` returns the trace as JSON or a 404.

* **Extensibility** – helper functions (`record_trace`, `update_trace_end`, `get_trace`) make it trivial to hook into any agent or tool call.

> * Truncation keeps the store small and logs readable.  
> * The middleware pattern keeps the rest of the codebase clean – no need to manually pass trace IDs everywhere.  
> * The implementation is framework‑agnostic: replace `warp` with Actix‑Web or Rocket by swapping the filter/route definitions.

    output: impl Into<String>,

        output: truncate(&output.into()),

pub struct Trace {

    pub fn add_step(&self, trace_id: &str, step: TraceStep) {
        let mut traces = self.traces.write().unwrap();

            steps: Vec::new(),

        trace.steps.push(step);
    }

    pub fn get_trace(&self, trace_id: &str) -> Option<Trace> {
        let traces = self.traces.read().unwrap();

    }
}

## 1. Acceptance Criteria
1. **Trace‑ID generation & propagation** – Every tool invocation creates a UUID‑v4 `trace_id`. The same `trace_id` is attached to the request context and **must not change** across any retry attempts of that logical call.
2. **Trace record fields** – Each stored trace entry contains exactly:
   - `trace_id` (string, UUID‑v4)  
   - `agent_id` (string)  
   - `tool_name` (string)  
   - `input` (JSON‑escaped, **truncated to 256 bytes** if longer)  
   - `output` (JSON‑escaped, **truncated to 256 bytes** if longer)  
   - `timestamp_start` (ISO‑8601, UTC)  
   - `timestamp_end` (ISO‑8601, UTC)  
3. **Structured logging** – On completion of a tool call the server writes **one** line to the JSON log with the fields above plus `"level":"trace"` and `"message":"tool call"`; the line must be valid JSON.
4. **In‑memory trace store** – The process‑wide store holds up to **10 000** most‑recent entries (FIFO eviction). Retrieval by `trace_id` returns **all** entries that share that ID, ordered by `timestamp_start`.
5. **GET `/traces/<trace_id>` – success** – Returns **HTTP 200** with a JSON object:
   ```json
   {
     "trace_id": "<trace_id>",
     "entries": [ { …trace entry… }, … ]
   }
   ```
   The `entries` array is sorted ascending by `timestamp_start`.
6. **GET `/traces/<trace_id>` – not found** – Returns **HTTP 404** with payload:
   ```json
   { "error": "trace not found" }
   ```
7. **Performance** – Under a load of **500 concurrent GET** requests for an existing `trace_id`, the 95th‑percentile response time must be **≤ 50 ms** (measured on a typical CI runner with 2 vCPU, 4 GB RAM).
